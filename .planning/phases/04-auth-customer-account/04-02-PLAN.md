<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → Supabase Auth API | Email + password cross this boundary in signUp/signIn |
| Alpine store → Nav DOM | Auth state (first name) rendered via x-text — XSS risk if unescaped |
| window.elvoraSignOut | Global function callable by any script on the page |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-02-01 | Information Disclosure | signIn error messages | mitigate | Throw Supabase error as-is; auth.html (Plan 03) maps error message to generic "Incorrect email or password" — no email enumeration from JS layer |
| T-04-02-02 | Tampering | x-text binding for first name in nav | mitigate | Alpine x-text escapes HTML automatically — never use x-html for user data (per RESEARCH security notes) |
| T-04-02-03 | Elevation of Privilege | client-side auth redirect in account.html | accept | RLS is the real gate; client redirect is UI convenience only; even if bypassed, Supabase RLS returns empty data |
| T-04-02-04 | Spoofing | window.elvoraSignOut callable by 3rd-party scripts | accept | Triggering signOut can only help security, not harm it; no sensitive operation performed |
| T-04-02-05 | Tampering | user_profiles upsert after signUp | mitigate | Catch profile upsert error separately; auth success not blocked by profile failure; RLS insert_own policy (auth.uid() = id) enforced at DB |
| T-04-02-SC | Tampering | npm/pip/cargo installs | accept | No new packages; all CDN deps already in project |
</threat_model>
