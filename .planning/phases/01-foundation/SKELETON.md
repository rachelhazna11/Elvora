# Elvora — Walking Skeleton

**Phase:** 01-Foundation
**Created:** 2026-06-10
**Purpose:** Architectural backbone — decisions locked here that all subsequent phases build on without renegotiation.

---

## What the Walking Skeleton Delivers

The thinnest possible end-to-end stack: a deployed Netlify site connected to a live Supabase project with a full database schema, seeded product catalog, storage buckets, and a working (stubbed) Edge Function. No UI beyond page shells.

A user opening the deployed URL sees an empty page. The Supabase SDK client in `js/supabase.js` can already query real product data. The `style-match` Edge Function is deployed and rejects unauthenticated requests.

## Architectural Decisions (Locked)

### Frontend Framework
| Decision | Value | Rationale |
|----------|-------|-----------|
| UI framework | Alpine.js 3.x (CDN) | No build step; declarative x-data in HTML; Alpine.store() for cart/auth state; 15 KB; ideal for multi-page assessment scope |
| JS modules | Vanilla ES modules via `<script type="module">` | Native browser support; no bundler needed for Phase 1–4; Vite added if complexity requires |
| CSS framework | Tailwind CSS v4 CLI | Utility-first; zero opinion on design; 5–20 KB production bundle after tree-shaking; luxury brands use utility-first for full visual control |

### CSS Architecture
| Decision | Value |
|----------|-------|
| Source file | `src/input.css` |
| Output file | `css/style.css` (gitignored; built by Tailwind CLI on Netlify) |
| Build command | `npx @tailwindcss/cli@next -i src/input.css -o css/style.css --minify` |
| Design tokens | CSS custom properties in `@theme` + mirrored in `:root` |
| Colour palette | Sage `#A8BFA3`, Rose `#D8A7A7`, Beige `#F5F1ED`, Charcoal `#2E2E2E`, Lavender `#C6B7E2`, Sage-light `#E8F0E6`, Rose-light `#F5E8E8`, Text-muted `#8A8A8A` |
| Typography | Playfair Display (serif, display/headlines) + Poppins (sans-serif, body/UI) via Google Fonts CDN |

### Backend
| Decision | Value | Rationale |
|----------|-------|-----------|
| Backend service | Supabase only | No custom server; Supabase Auth + DB + Storage + Edge Functions covers all backend needs |
| Database | Supabase PostgreSQL | Full relational DB; RLS enforces access control |
| Auth | Supabase Auth email/password | No separate auth library; JWT auto-refresh built in |
| Storage | Supabase Storage | `product-images` (public CDN), `user-uploads` (private, user-scoped) |
| Edge Functions | Supabase Edge Functions (Deno) | API key proxy for Gemini Vision; keeps `GEMINI_API_KEY` out of browser |

### Security Architecture
| Decision | Value |
|----------|-------|
| RLS | Enabled on ALL 16 tables — non-negotiable |
| Frontend key | Only `SUPABASE_ANON_KEY` exposed in JS (safe by design; RLS enforces access) |
| Service role key | Never in any frontend file or git history — Supabase dashboard use only |
| Gemini API key | Supabase Edge Function secrets only — never in HTML, JS, .env, or git |
| Admin role | Stored in `auth.users.raw_app_meta_data.role = 'admin'`; checked via `is_admin()` security-definer function |

### AI Integration
| Decision | Value |
|----------|-------|
| AI model | Gemini Vision (Phase 5; stub in Phase 1) |
| Proxy pattern | Supabase Edge Function `style-match` — browser never calls Gemini directly |
| API key location | Supabase Edge Function secret `GEMINI_API_KEY` |
| Phase 1 stub | Returns hardcoded mock response matching Phase 5 contract exactly |

### Deployment
| Decision | Value |
|----------|-------|
| Hosting | Netlify (free tier) |
| Functions | Supabase Edge Functions (not Netlify Functions) |
| CI/CD | Netlify auto-deploy from GitHub main branch |
| Keep-alive | GitHub Actions cron (weekly) pinging Supabase `/auth/v1/health` |
| Dev/prod split | Single Supabase project (free tier constraint) |
| Schema management | Cloud-only — SQL files in `supabase/migrations/`, applied manually via Supabase SQL Editor |

## Data Model Summary

16 tables in PostgreSQL:

| Table | Purpose | RLS |
|-------|---------|-----|
| user_profiles | Extended user data (name, style preferences) | User-scoped |
| categories | Product activity categories (leggings, sports-bra, etc.) | Public read / admin write |
| products | Product catalog | Public read / admin write |
| product_variants | Per-product colour + size combinations | Public read / admin write |
| product_images | Product image URLs (Unsplash CDN for seed) | Public read / admin write |
| collections | Curated product collections (Best Sellers, Padel Edit) | Public read / admin write |
| collection_products | Many-to-many products ↔ collections | Public read / admin write |
| cart_items | Shopping cart (authenticated users) | User-scoped |
| wishlist_items | Saved products | User-scoped |
| orders | Order header | User-scoped + guest email |
| order_items | Snapshotted order lines (unit_price, product_name, variant_label) | Order-scoped |
| reviews | Product reviews (admin-seeded v1) | Public read / admin write |
| testimonials | Homepage social proof | Public read (active only) / admin write |
| newsletter_subscribers | Email signups | Public insert / admin read |
| ai_style_sessions | Style Match history | User-scoped + guest token |
| promo_codes | Discount codes (stub) | Public read (active) / admin write |

## Invariants — Future Phases Must Not Violate

1. **Never re-initialize Supabase client** — import from `js/supabase.js` only. One client, one place.
2. **Never add `SERVICE_ROLE_KEY` to any frontend file** — it bypasses RLS entirely.
3. **Never disable RLS on any table** — security model depends on all 16 tables having RLS.
4. **Never call Gemini API directly from the browser** — always proxy via the `style-match` Edge Function.
5. **No inline `<script>` blocks in HTML** — all JS in `js/*.js` ES module files (NF-006).
6. **CSS compiled by Tailwind CLI, not Play CDN** — `css/style.css` is the build output; Play CDN is forbidden in production.
7. **is_admin() reads `app_metadata.role`** (not `user_metadata`) — user_metadata is user-editable; app_metadata is service-role-only.
