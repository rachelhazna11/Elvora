<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| visitor → Supabase newsletter_subscribers | Email submitted by visitor crosses browser → Supabase boundary; anon key used |
| Alpine inline x-data → supabase global | Module-scoped supabase client exposed to window for Alpine inline components |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-08 | Denial of Service | newsletter_subscribers INSERT | mitigate | Postgres unique constraint on email column (already in schema) prevents duplicate rows. Rate limiting at Netlify/Supabase infrastructure level; not Phase 2 code scope. |
| T-02-09 | Tampering | window.supabase global exposure | accept | Exposing the supabase client to window with the anon key is equivalent to it being in browser code — safe by design because RLS restricts what anon can do. Only permitted operations (INSERT on newsletter_subscribers) are allowed by RLS. |
| T-02-10 | Information Disclosure | Supabase error messages in console | accept | Error messages are caught and replaced with user-friendly strings; raw Supabase error objects are not displayed to users |
| T-02-11 | Input Validation — newsletter email | V5 | mitigate | HTML5 type="email" provides client-side format validation. Alpine checks non-empty + includes '@' before calling Supabase. Server-side: Postgres email CHECK constraint in schema (if present) provides final validation. |
| T-02-SC | Tampering | npm/pip/cargo installs | accept | No new packages installed in this plan |
</threat_model>
